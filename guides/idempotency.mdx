---
title: "Idempotency"
description: "How to retry money-moving operations safely. The Idempotency-Key header on /fund, externalId on trades."
---

Anything that moves money is safe to retry **only** if the server can
recognise the retry as a duplicate. SkinShark gives you two mechanisms,
depending on the endpoint.

## Idempotency-Key header (on funding)

`POST /merchant/users/{id}/fund` requires an `Idempotency-Key` header.
Replays with the same key — for the same merchant — return the original
transaction with `idempotent: true`, regardless of whether the body
matches.

```http
POST /merchant/users/user_42/fund
api-key: sk_live_...
Idempotency-Key: 01h0bf2c-e9d8-4d3a-9c70-...
content-type: application/json

{ "amount": "25.00" }
```

```json
// First call
{ "transactionId": "01h0...", "idempotent": false }

// Replay with same key
{ "transactionId": "01h0...", "idempotent": true }
```

<Tip>
Generate the key once per logical operation, not per HTTP attempt. If your
checkout retries on network errors, all retries should send the same key.
</Tip>

```ts
import { randomUUID } from "node:crypto";

async function fundForCheckout(checkoutId: string, subUserId: string, amount: string) {
  // Derive deterministically from checkoutId so retries from anywhere in
  // your system reuse the same key.
  const idempotencyKey = `checkout:${checkoutId}`;

  return api<{ transactionId: string; idempotent: boolean }>(
    `/merchant/users/${subUserId}/fund`,
    {
      method: "POST",
      headers: { "idempotency-key": idempotencyKey },
      body: JSON.stringify({ amount }),
    },
  );
}
```

### Scope

Idempotency keys are scoped per merchant: `merchant:<merchantId>:<key>`.
Two different merchants can use the same key without colliding, and a
sub-user's externalId never affects scope.

### Errors

| Code | Key | Meaning |
|---|---|---|
| 1705 | `IDEMPOTENCY_KEY_REQUIRED` | Header missing on `/fund`. |
| 1704 | `IDEMPOTENCY_CONFLICT` | Same key, different body — server returns the original transaction. |

## externalId (on trades and sub-users)

`POST /market/buy`, `POST /market/buy/quick`, and `POST /merchant/users`
accept an `externalId` field in the body. It's stored under a uniqueness
constraint:

- **Sub-users:** unique per `(merchantId, externalId)`.
- **Trades:** unique per `(subUserId, externalId)`.

This means a *correct* retry — same `externalId`, same sub-user, same
intent — is rejected at the database layer rather than creating a
duplicate. **Unlike `/fund`, `/market/buy` is not transparently idempotent**:
you don't get the original trade back. Instead you get a `CONFLICT`
(1500), and you reconcile by looking up the original.

```ts
async function buyOrReconcile(input: {
  externalId: string;
  subUserId: string;
  items: BuyItem[];
}): Promise<Trade> {
  try {
    return await api<Trade>("/market/buy", {
      method: "POST",
      headers: { "on-behalf-of": input.subUserId },
      body: JSON.stringify({ items: input.items, externalId: input.externalId }),
    });
  } catch (err) {
    if (err instanceof SkinSharkError && err.key === "CONFLICT") {
      // Look up by your own externalId
      const found = await api<{ items: Trade[] }>(
        `/merchant/trades?externalId=${encodeURIComponent(input.externalId)}&limit=1`,
      );
      if (found.items[0]) return found.items[0];
    }
    throw err;
  }
}
```

Use `externalId` as your join key everywhere — when you reconcile
SkinShark trades back into your own order table, you look them up by your
own checkout ID, not the SkinShark UUID.

### Picking values

Pick `externalId` values that uniquely identify a *logical* operation —
don't recycle them across different intents. A retried checkout reuses
the same ID; a fresh checkout gets a fresh ID.

## Operations without idempotency

Some endpoints don't have idempotency support yet:

- Most catalog reads (search, listings, item detail) — these are GETs and
  free to retry.
- Deposit `quote` and `session` endpoints — quotes are short-lived; if
  you retry within the TTL you'll get a different quote. Don't retry on
  network failure; create a fresh quote.

For deposit `create` paths (`POST /user/wallet/deposit/gate`,
`/onramp/session`), retrying after a failure can produce two pending
deposits. Pattern: walk recent deposits for the sub-user and resume the
existing one rather than creating fresh.

```ts
try {
  return await api("/user/wallet/deposit/gate", { method: "POST", body }, subUserId);
} catch (err) {
  if (err instanceof SkinSharkError && err.key === "INVALID_QUOTE") {
    // Quote expired — re-quote, don't replay
    const quote = await api("/user/wallet/deposit/gate/quote", { method: "POST", body: requote }, subUserId);
    return api("/user/wallet/deposit/gate", { method: "POST", body: { ...body, quoteToken: quote.quoteToken } }, subUserId);
  }
  throw err;
}
```

Use `POST /user/wallet/deposit/{id}/resume` to fetch a fresh hosted URL
or pay-to address for an existing deposit.

## When to retry vs reconcile

| Failure mode | Action |
|---|---|
| Network timeout | Retry with the same idempotency key. |
| `5xx` from SkinShark | Retry with the same idempotency key. |
| `4xx` other than `429` | Don't retry. Fix the input. |
| `429 Rate limited` | Honour `Retry-After`, then retry. |
| Truly unknown state (e.g. you crashed mid-call) | Reconcile: query `/merchant/trades?externalId=<your-id>` to see if the trade made it. |
