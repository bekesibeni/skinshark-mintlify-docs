You don't need an SDK to use the API — `fetch` plus the typed wrapper
from [Authentication](/guides/authentication) covers everything. For
fully typed responses with zero hand-maintenance, generate a `paths`
type from the OpenAPI spec.

## Generate types from the spec

The OpenAPI spec is the source of truth. Generate a `paths` type with
[`openapi-typescript`](https://github.com/openapi-ts/openapi-typescript)
and you get full IntelliSense over every endpoint.

```bash
npm install -D openapi-typescript
npx openapi-typescript \
  https://raw.githubusercontent.com/skinshark/skinshark-api/main/docs/skinshark-merchant-api-openapi.yaml \
  -o src/skinshark-types.ts
```

```ts
import type { paths } from "./skinshark-types";

type CreateSubUserBody =
  paths["/merchant/users"]["post"]["requestBody"]["content"]["application/json"];

type CreateSubUserResponse =
  paths["/merchant/users"]["post"]["responses"]["201"]["content"]["application/json"]["data"];

type ListTradesQuery = NonNullable<
  paths["/merchant/trades"]["get"]["parameters"]["query"]
>;
```

## A typed fetch helper

Pair it with a tiny `fetch` wrapper that infers the `data` shape from the
operation's success response:

```ts
import type { paths } from "./skinshark-types";

type Method = "get" | "post" | "patch" | "delete";

type SuccessData<P extends keyof paths, M extends Method> =
  paths[P][M] extends { responses: { 200: { content: { "application/json": { data: infer D } } } } }
    ? D
    : paths[P][M] extends { responses: { 201: { content: { "application/json": { data: infer D } } } } }
      ? D
      : never;

export async function call<P extends keyof paths, M extends Method>(
  method: M,
  path: P,
  init?: { body?: unknown; onBehalfOf?: string; query?: Record<string, string>; idempotencyKey?: string },
): Promise<SuccessData<P, M>> {
  const qs = init?.query
    ? "?" + new URLSearchParams(init.query).toString()
    : "";

  const headers: Record<string, string> = {
    "api-key": process.env.SKINSHARK_API_KEY!,
    "content-type": "application/json",
  };
  if (init?.onBehalfOf) headers["on-behalf-of"] = init.onBehalfOf;
  if (init?.idempotencyKey) headers["idempotency-key"] = init.idempotencyKey;

  const res = await fetch(`https://api.skinshark.gg${path}${qs}`, {
    method: method.toUpperCase(),
    headers,
    body: init?.body ? JSON.stringify(init.body) : undefined,
  });

  const env = await res.json();
  if (!env.success) {
    const e = env.error;
    throw new SkinSharkError(env.requestId, e.code, e.key, e.message);
  }
  return env.data;
}
```

```ts
const profile = await call("get", "/merchant");
//    ^? { id: string; email: string; merchantFeeBps: number; ... }

const sub = await call("post", "/merchant/users", {
  body: { email: "alice@example.com", externalId: "user_42" },
});
//    ^? { id: string; externalId: string | null; ... }

const trades = await call("get", "/merchant/trades", {
  query: { status: "completed", limit: "50" },
});
//    ^? { items: Trade[]; nextCursor: string | null }
```

This is the highest-leverage option: zero runtime dependencies, types
drift automatically as the spec evolves, and the wrapper is ~30 lines.
Pin the spec to a git tag if you want stable types between releases.

## Types only, no client

If you already have an HTTP client you're happy with, skip the wrapper
and just consume the generated `paths` type:

```ts
import type { paths } from "./skinshark-types";

type Trade = paths["/merchant/trades/{id}"]["get"]["responses"][200]["content"]["application/json"]["data"];
type FundBody = paths["/merchant/users/{id}/fund"]["post"]["requestBody"]["content"]["application/json"];
```

For shorter aliases, hand-roll the most-used types in
[Data models](/reference/data-models) (or copy them from there) — those
mirror the spec but read better in code.

## Webhook signature verification

Drop the `verifySkinSharkSignature` helper from
[Webhooks](/guides/webhooks#signature-verification) into your codebase.
It's ~30 lines, has no dependencies beyond `node:crypto`, and follows
the [Standard Webhooks](https://www.standardwebhooks.com/) scheme — so
any Standard Webhooks library on the npm side works just as well.
