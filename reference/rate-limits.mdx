Rate limits are enforced per merchant API key on a sliding window. Limits
are documented per-endpoint in the [API reference](/api-reference) — this
page summarises the patterns and the response shape.

## Limits by endpoint group

The values below come from the route handlers' `rateLimit.config`. Treat
them as the contract; if a real-world spike pushes you against them, file
a request to raise the cap rather than working around it client-side.

| `POST /merchant/users` (create) | 1 min | 100 |

| `POST /merchant/users/{id}/suspend` | 1 min | 30 |
| `POST /merchant/users/{id}/reactivate` | 1 min | 30 |
| `DELETE /merchant/users/{id}` | 1 min | 30 |

for normal traffic).

## Response headers

Every response includes:

| Header | Meaning |
|---|---|
| `x-ratelimit-limit` | Window cap. |
| `x-ratelimit-remaining` | Calls left in the current window. |
| `x-ratelimit-reset` | Epoch seconds at which the window resets. |

When you hit the limit, the response is `429` with:

| Header | Meaning |
|---|---|
| `retry-after` | Seconds to wait before retrying. |
| `x-ratelimit-remaining: 0` | |
| `x-ratelimit-reset` | Epoch seconds. |

```json
{
  "requestId": "01900b...",
  "success": false,
  "error": {
    "code": 1600,
    "key": "RATE_LIMITED",
    "message": "Too many requests"
  }
}
```

## Backing off

```ts
async function withRateLimitRetry<T>(fn: () => Promise<Response>, body: () => Promise<T>): Promise<T> {
  for (let i = 0; i < 5; i++) {
    const res = await fn();
    if (res.status !== 429) return body();

    const retryAfterHeader = res.headers.get("retry-after");
    const seconds = retryAfterHeader ? Number(retryAfterHeader) : 1;
    await new Promise((r) => setTimeout(r, seconds * 1000));
  }
  throw new Error("rate limit retry exhausted");
}
```

## Per-IP limits

Beyond per-key limits, the platform applies a per-IP guard against
brute-forcing auth endpoints. If you're calling from a single egress IP
across many merchant keys (e.g. a multi-tenant proxy), you may hit IP
limits before per-key limits — split egress IPs or contact support.

## Request ID

Every response (success or 429) includes the `x-request-id` header (also
echoed in the envelope `requestId`). Include it in any rate-limit
support request — it lets us trace the exact denial.
